# Bug Report: v0.7 Autorun Process Critical Issues

**Date**: 2026-02-04
**Scope**: Autorun three-stage workflow in v0.7 codebase (main.py is deprecated v0.6.1)
**Severity**: CRITICAL - Multiple KeyError crashes will occur during autorun

---

## Executive Summary

The v0.7 migration renamed stage confirmation keys from `stage<n>_confirmation` to `stage<n>_message`, but **5 critical code paths were not updated**. This will cause:
1. **KeyError crashes** when template formatting fails
2. **is_premature_stop()** will fail to detect completion markers
3. **ai_monitor** will fail to recognize stop markers
4. **Fallback CONFIG** in tests will use wrong values

---

## CRITICAL BUG #1: is_premature_stop() Uses Non-Existent Keys

**File**: `plugins/clautorun/src/clautorun/plugins.py:575-577`

**Current Code**:
```python
return not any(marker in combined for marker in [
    CONFIG["stage1_confirmation"],  # ← KeyError: 'stage1_confirmation'
    CONFIG["stage2_confirmation"],  # ← KeyError: 'stage2_confirmation'
    CONFIG["stage3_confirmation"],  # ← KeyError: 'stage3_confirmation'
    CONFIG.get("completion_marker", ""),
    CONFIG["emergency_stop"]
])
```

**Impact**:
- **CRASH**: KeyError will occur on first autorun Stop hook call
- is_premature_stop() will never execute, causing infinite injection loops

**Fix**:
```python
return not any(marker in combined for marker in [
    CONFIG["stage1_message"],       # ✅ Correct key
    CONFIG["stage2_message"],       # ✅ Correct key
    CONFIG["stage3_message"],       # ✅ Correct key
    CONFIG.get("completion_marker", ""),
    CONFIG["emergency_stop"]
])
```

**Test Coverage**: Should add test for `test_is_premature_stop_checks_correct_keys`

## EDGE CASES AND FAILURE MODES

### Edge Case #1: Premature Stage 3 Output During Countdown

**Location**: `plugins.py:812-820`

**Current Behavior**:
```python
if CONFIG["stage3_message"] in combined:
    if remaining > 0:
        ctx.autorun_stage = EventContext.STAGE_2  # ← Resets to STAGE_2
        return inject(f"Too early for Stage 3. Continue Stage 2...")
    else:
        ctx.autorun_active = False  # ← Stops autorun
        return None
```

**Edge Case**: What if AI outputs Stage 3 message with `remaining == 0` but work is incomplete?

**Risk**: System stops autorun even if tasks aren't actually complete

**Mitigation**: This is by design - countdown is a "cooling off" period to prevent premature completion. AI should only output stage3_message when truly done.

**Test Coverage**: Add `test_premature_stage3_during_countdown_resets_to_stage2`

### Inconsistency #2: Dual-Key Pattern Not Fully Implemented

**Location**: Multiple files

**Plan Specification**: Each stage should have TWO keys:
- `stage<n>_completion` - Text injected TO AI (what stage means)
- `stage<n>_message` - ALL-CAPS string AI outputs BACK (confirmation)

**Current Implementation**:
- `config.py` defines both `stage<n>_completion` and `stage<n>_message` ✅
- But `stage<n>_completion` is NEVER USED in code ❌
- Only `stage<n>_message` and `stage<n>_instruction` are used

**Impact**:
- `stage<n>_completion` keys are dead code
- Plan specification not followed

**Question**: Should we:
- A. Remove unused `stage<n>_completion` keys?
- B. Use them in progressive disclosure?

**Recommendation**: Option B - Use in progressive disclosure base_template
