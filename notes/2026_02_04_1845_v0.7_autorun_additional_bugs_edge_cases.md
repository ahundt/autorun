## CRITICAL BUG #6: Emergency Stop Doesn't Actually Stop

**Severity**: CRITICAL
**File**: `plugins/clautorun/src/clautorun/plugins.py:755-833` (autorun_injection)

### Problem

When AI outputs `CONFIG["emergency_stop"]` string ("AUTORUN_STATE_PRESERVATION_EMERGENCY_STOP"), the system **continues running** instead of stopping.

### Root Cause

1. `is_premature_stop()` (line 565-580) checks if emergency_stop is in transcript
2. If present, `is_premature_stop()` returns **False** (line 574-579)
3. This prevents the "premature stop" warning from triggering
4. But **no code path explicitly checks for emergency_stop and stops autorun**
5. Function falls through to fallback (line 832) and keeps injecting prompts

### Code Flow When Emergency Stop Occurs

```python
# Line 769: combined = result + transcript  (contains emergency_stop)

# Line 780-792: Stage 1 checks
if stage == EventContext.STAGE_1:
    if CONFIG["stage1_message"] in combined:  # False - no stage1 message
        ...
    if CONFIG["stage3_message"] in combined:  # False - no stage3 message
        ...
    if is_premature_stop(ctx):  # False - emergency_stop is in combined!
        ...  # This block doesn't run

# Doesn't return - falls through!

# Line 832: Fallback
ctx.recheck_count = (ctx.recheck_count or 0) + 1
return inject(build_injection_prompt(ctx))  # ← BUG: Keeps running!
```

### Expected Behavior

When emergency_stop is detected, autorun should:
1. Immediately deactivate: `ctx.autorun_active = False`
2. Reset stage: `ctx.autorun_stage = EventContext.STAGE_INACTIVE`
3. Stop external monitor: `stop_external_monitor(ctx)`
4. Return `None` to allow Claude to stop

### Fix

Add explicit emergency stop check at the beginning of autorun_injection():

```python
def autorun_injection(ctx: EventContext) -> Optional[Dict]:
    if not ctx.autorun_active:
        return None

    result = ctx.tool_result or ""
    transcript = ctx.transcript.text
    combined = result + transcript

    # === EMERGENCY STOP CHECK (NEW) ===
    if CONFIG["emergency_stop"] in combined:
        ctx.autorun_active = False
        ctx.autorun_stage = EventContext.STAGE_INACTIVE
        stop_external_monitor(ctx)
        return None  # Allow Claude to stop - emergency stop detected

    stage = ctx.autorun_stage
    # ... rest of function
```

### Impact

**Current**: Emergency stop string is documented and injected in templates, but does nothing. AI thinks it can trigger emergency stop, but system ignores it and keeps running.

**Users affected**: Anyone who encounters high-risk situations where AI correctly outputs emergency stop string. System will dangerously continue instead of stopping.

### Test Coverage

Need test: `test_emergency_stop_immediately_halts_autorun`

```python
def test_emergency_stop_immediately_halts_autorun():
    ctx = EventContext(session_id="test", event="Stop")
    ctx.autorun_active = True
    ctx.autorun_stage = EventContext.STAGE_1
    ctx.tool_result = CONFIG["emergency_stop"]

    result = autorun_injection(ctx)

    assert result is None, "Should return None to allow stop"
    assert ctx.autorun_active == False, "Should deactivate autorun"
    assert ctx.autorun_stage == EventContext.STAGE_INACTIVE, "Should reset stage"
```

## BUG #7: Missing Stage Transition Validation

**Severity**: MEDIUM
**File**: `plugins/clautorun/src/clautorun/plugins.py:780-806`

### Problem

AI can output stage markers out of order, but the system doesn't validate all transitions:

1. **Stage 1**: Checks for premature stage3_message (line 786), but NOT for premature stage2_message
2. **Stage 2**: Checks for premature stage3_message (line 801), but NOT for regressing to stage1_message

### Missing Checks

**In Stage 1** (should add):
```python
if CONFIG["stage2_message"] in combined:
    return inject(f"Complete Stage 1 first. Output **{CONFIG['stage1_message']}** when done.")
```

**In Stage 2** (should add):
```python
if CONFIG["stage1_message"] in combined:
    return inject(f"Already in Stage 2. Output **{CONFIG['stage2_message']}** when complete.")
```

### Current Behavior

If AI outputs wrong stage marker:
- Falls through to `is_premature_stop()`
- Returns False (marker is in the list)
- Falls through to fallback
- Re-injects prompt and continues

**Result**: Confusing - AI outputs a completion marker, but nothing happens and it gets more instructions.

### Expected Behavior

System should explicitly warn AI about out-of-order stage markers.

### Impact

**Low frequency**: AI rarely outputs wrong stage marker
**Medium severity**: When it happens, creates confusion and wasted tokens

### Test Coverage

Need tests:
- `test_premature_stage2_message_in_stage1_warns`
- `test_stage1_message_in_stage2_warns_about_regression`

## BUG #8: Countdown Hook Count Reset Timing Issue

**Severity**: LOW
**File**: `plugins/clautorun/src/clautorun/plugins.py:773, 797`

### Problem

Countdown tracking has off-by-one error:

1. Line 773: `ctx.hook_call_count = (ctx.hook_call_count or 0) + 1` - Increment at function start
2. Line 797: `ctx.hook_call_count = 0` - Reset when entering STAGE_2_COMPLETED
3. Line 810: `remaining = countdown_max - ctx.hook_call_count` - Calculate remaining

**Issue**: When transitioning to STAGE_2_COMPLETED:
- Line 797 resets hook_call_count to 0
- Function returns
- Next hook call: Line 773 increments to 1 (not 0!)
- Line 810: `remaining = 3 - 1 = 2` (expected: 3)

**Result**: Countdown is 1 call shorter than configured (shows 2 remaining instead of 3).

### Fix Options

**Option A**: Don't reset to 0, reset to -1:
```python
ctx.hook_call_count = -1  # Next increment makes it 0
```

**Option B**: Increment after countdown check, not at start:
```python
# Move line 773 increment to end of function
# Or only increment in countdown section
```

**Option C**: Use separate countdown counter:
```python
ctx.countdown_calls = 0  # Separate from hook_call_count
```

### Impact

**Low**: Countdown is 1 shorter than configured (2 instead of 3)
**User-visible**: Yes - countdown messages show wrong numbers

### Test Coverage

Need test: `test_countdown_shows_correct_remaining_count`

## BUG #9: ExitPlanMode Gate Checks Transcript Instead of Stage

**Severity**: MEDIUM
**File**: `plugins/clautorun/src/clautorun/plugins.py:144-145`

### Problem

Gate checks if stage3_message is **in transcript**, not if **current session reached Stage 3**:

```python
transcript = ctx.transcript.text
if CONFIG["stage3_message"] not in transcript:
    return ctx.deny("Cannot exit plan mode yet...")
```

**Issue**: If stage3_message appears in transcript from:
- Previous autorun session in same Claude session
- User manually typed it
- Earlier in conversation (before current autorun started)

Then ExitPlanMode would be allowed even though **current** autorun hasn't reached Stage 3.

### Expected Behavior

Gate should check **both**:
1. Transcript contains stage3_message (current check)
2. Current autorun_stage >= STAGE_2_COMPLETED (new check)

### Fix

```python
transcript = ctx.transcript.text
stage = ctx.autorun_stage

# Check both transcript AND current stage
if CONFIG["stage3_message"] not in transcript:
    return ctx.deny("Cannot exit plan mode yet. Complete plan verification first...")

# NEW: Also verify we're actually in Stage 3 (not leftover from previous session)
if stage != EventContext.STAGE_2_COMPLETED:
    return ctx.deny(
        f"Stage 3 not yet reached in current session.\n"
        f"Current stage: {stage}\n"
        f"Complete Stage 1 and Stage 2 before exiting plan mode."
    )

return None  # Allow - both checks pass
```

### Impact

**Low frequency**: Rare that stage3_message appears from previous session
**High severity when occurs**: User could exit plan without verification

### Test Coverage

Need test: `test_exit_plan_mode_gate_checks_current_stage_not_just_transcript`

prompt = ctx.activation_prompt or ctx.prompt or ""  # Ensure string

## EDGE CASE #4: Countdown Can Be Negative

**Severity**: LOW
**File**: `plugins/clautorun/src/clautorun/plugins.py:810`

### Problem

```python
remaining = countdown_max - ctx.hook_call_count
```

If `hook_call_count` somehow exceeds `countdown_max`, remaining becomes negative.

**Line 822**: `if remaining > 0:` handles positive case
**Line 828**: `else:` handles 0 and negative cases

**Current behavior**: Negative countdown is treated same as 0 (reveals Stage 3).

### Expected Behavior

Probably fine, but should document or add assertion.

### Fix (Optional)

```python
remaining = max(0, countdown_max - ctx.hook_call_count)  # Clamp to 0
```

## EDGE CASE #5: Stage Transition Without Marker Check

**Severity**: LOW
**File**: Multiple stage transition points

### Problem

Stage transitions check for presence of markers but don't validate **state consistency**:

1. What if stage1_message appears but Stage 1 work isn't actually done?
2. What if transcript is truncated and marker is lost?
3. What if marker appears from previous session context?

### Mitigation

This is by design - system trusts AI's output. If AI outputs marker prematurely:
- Next Stop hook will detect incomplete work
- is_premature_stop() or stage-specific checks will re-inject
- Forced compliance template eventually triggers
