You are reviewing a Python asyncio daemon (clautorun) that acts as a shared hook server for Claude Code and Gemini CLI sessions running concurrently. The daemon has a critical bug: its asyncio event loop permanently blocks when a handler spins at 98.5% CPU, causing all subsequent hook connections to time out.

## Context: What the storage is used for

The daemon tracks two small dicts per project:
1. `active_plans`: {plan_file_path_str -> metadata_dict} — plans currently in progress
2. `tracking`: {content_hash -> {exported_to, exported_at}} — plans already exported

These are written on ExitPlanMode (accepted/abandoned plans) and read on SessionStart (recovery). Multiple Claude Code sessions + Gemini CLI sessions connect to the same daemon concurrently via Unix socket.

## Current Implementation (the problem)

`session_manager.py` is ~420 lines implementing custom RAII locking around Python's `shelve`:
- `SessionLock` class: uses `fcntl.flock(LOCK_EX | LOCK_NB)` with retry loop (100 retries × 0.01s = ~1s timeout)
- `_lock_registry`: threading.Lock-guarded dict keyed by (pid, thread_id) for reentrant detection
- `SessionBackendManager`: tests default/dumbdbm/memory backends at startup
- `session_state()` contextmanager: acquires flock → opens shelve → yields → closes shelve → releases flock

**The critical bugs this causes:**
1. When `export_on_exit_plan_mode()` is called (PostToolUse handler), the daemon spins at 98.5% CPU and never returns — the asyncio event loop is completely blocked indefinitely
2. FD leak: the lock file FD and 50MB shelve FD are never released (context manager's `__exit__` never called because handler never returns)
3. macOS flock non-reentrance: same process, different fd → errno=35 (EAGAIN) even with `_lock_registry` protection
4. `handle_client()` in core.py calls `self.app.dispatch(ctx)` **synchronously** — no `await`, no timeout — so any handler hang blocks the entire event loop
